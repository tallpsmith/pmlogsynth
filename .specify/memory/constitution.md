# pmlogsynth Constitution

## Core Principles

### I. PCP Archive Fidelity (NON-NEGOTIABLE)

Every archive generated by `pmlogsynth` MUST be indistinguishable in format from
archives produced by `pmlogger`. Specifically:

- Archives MUST pass `pmlogcheck` without errors or warnings for all supported
  hardware profiles and all valid stressor combinations.
- All PCP metric identifiers, units, and semantics MUST match those defined in the
  installed PCP PMDA — no invented names, no invented units.
- Internal metric consistency constraints MUST be enforced at every sample:
  `user + sys + idle + wait + steal == total_ticks` per CPU; `used + free == physmem`.
- Counter metrics MUST accumulate monotonically from zero; noise MUST NOT produce
  negative counter increments.
- Archive output MUST be usable with `pmval`, `pmrep`, `pmlogdump`, and `pmlogsummary`
  without any post-processing.

Rationale: the tool's entire value proposition is that generated archives are valid PCP
archives. A single pmlogcheck failure or a metric naming divergence invalidates the tool
for its target use case (CI-based PCP tooling regression testing).

### II. Layered Testing (NON-NEGOTIABLE)

The test suite is structured in three tiers. Tier membership is a hard contract:

- **Tier 1 — Unit**: MUST run on any Python 3.8+ system with zero PCP packages installed.
  Covers profile loading, validation, timeline sequencing, phase transitions, repeat
  expansion, noise application, counter accumulation, and all domain consistency
  constraints — at value-computation level, no archive write required.
- **Tier 2 — Integration**: MUST run on any system (with or without PCP) by stubbing the
  `libpcp_import` / `pcp.pmi` layer. Verifies the full pipeline from parsed profile
  through to the writer call contract (correct metric names, values, timestamps).
- **Tier 3 — E2E**: Conditionally runs when `pcp.pmi` is importable. Generates a real
  archive, runs `pmlogcheck`, and asserts metric values via `pmval`. Detection MUST be
  by import attempt, not by PATH check.

Tier 1 and Tier 2 MUST run in CI on every push. A Tier 3 skip MUST emit a visible
warning, never a silent pass.

Rationale: Tier 1/2 without PCP installed enables fast iteration on any developer
machine. Tier 3 provides ground truth but cannot be the only gate — it would block
development on macOS where the apt PCP Python bindings are unavailable.

### III. Declarative Profile-First

The YAML workload profile is the authoritative input to the tool. All validation,
defaults, and constraint enforcement flow from the profile schema:

- Stressor field defaults MUST be applied at value-computation time (in `MetricModel`),
  not at parse time. Absent fields MUST remain `None` after parsing so that Phase 3
  overlay merge logic can distinguish "not set" from "set to default".
- The hardware profile schema (fields, types, required keys) is FROZEN after Phase 1
  ships. Any breaking change requires a schema version bump and a documented migration
  path, because Phase 2 and Phase 3 depend on it.
- The `host` block MUST support exactly three mutually exclusive forms: (1) named profile
  only; (2) named profile + explicit `overrides:` sub-block; (3) fully inline. Mixing
  bare inline fields with `host.profile` without an `overrides:` key is a validation
  error — this pattern is intentional and consistent with Phase 3 overlay semantics.

Rationale: the profile is the user-facing contract. Ambiguous defaults baked into parsed
objects make Phase 3 overlay merging impossible without a refactor.

### IV. Phase-Aware Extensibility

Phase 1 architecture decisions MUST NOT require refactoring in Phases 2 or 3. The
following are non-negotiable implementation choices that exist specifically to avoid
future breakage:

- `ProfileLoader` MUST expose both `from_file(path)` and `from_string(yaml_text)`; Phase
  2 uses `from_string` to pass Claude-generated YAML directly.
- The CLI MUST use argparse subparsers from day one; `fleet` is a reserved subcommand.
- `ValueSampler.__init__` MUST accept an optional `seed` parameter (default `None`);
  Phase 3 uses it for byte-identical reproducible fleet archives.
- The `-C / --config-dir` option MUST be implemented via a standalone `ProfileResolver`
  (or equivalent) that both the top-level command and the future `fleet` subcommand
  instantiate from the same CLI argument, without reimplementation.

Any deviation from these decisions MUST be justified in the Complexity Tracking section
of the implementation plan and reviewed before implementation.

### V. Minimal External Dependencies

The tool's core archive generation pipeline MUST NOT introduce dependencies beyond:

- Python 3.8+ standard library
- PCP installed on the host (`libpcp_import.so`, `python3-pcp`)
- PyYAML (the only permitted third-party pip dependency for core generation)

The following are explicitly prohibited for the core package:
- NumPy or any scientific Python stack (use `random.gauss` from stdlib)
- A running `pmcd` daemon
- Root access
- A C compiler

Optional extras (e.g., `[ai]` for Phase 2) are permitted as pip extras, not as default
dependencies.

Rationale: the tool must install and run in minimal CI environments. Adding NumPy pulls
in a 30 MB binary dependency for functionality that stdlib covers adequately.

### VI. CI-First Quality Gates

The same quality gates MUST run both in `pre-commit.sh` (local) and in the GitHub Actions
CI workflow. No gate may exist in one but not the other:

- Static analysis (linting) — MUST run unconditionally
- Static type checking — MUST run unconditionally
- Tier 1 unit tests — MUST run unconditionally
- Tier 2 integration tests — MUST run unconditionally
- Tier 3 E2E tests — run conditionally (skip with visible warning if PCP unavailable)

The GitHub Actions workflow MUST:
- Trigger on push to any branch and on pull requests targeting `main`
- Run static analysis, type checking, Tier 1, and Tier 2 in a matrix across Python 3.8
  and the latest stable Python release
- Run Tier 3 E2E separately on system Python (ubuntu-latest) against an apt-installed PCP

Incremental delivery MUST be enabled by keeping CI green on every push. The CI workflow
MUST be one of the first artifacts created so every subsequent push gets automated
feedback.

Rationale: quality gates that differ between local and CI create "works on my machine"
failures and erode trust in the gate. Front-loading CI setup enables incremental delivery
with continuous validation.

## Technology Stack

- **Language**: Python 3.8+ (minimum); latest stable tested in CI matrix
- **Archive writing**: `pcp.pmi.pmiLogImport` via the `python3-pcp` system package
- **Profile parsing**: PyYAML
- **Testing**: pytest; `unittest.mock` (stdlib) for Tier 2 PCP stubs
- **Linting**: ruff (covers flake8 + isort; fast, Python 3.8 compatible)
- **Type checking**: mypy (strict mode; Python 3.8 compatible)
- **Packaging**: `pyproject.toml` (PEP 517/518); `importlib.resources` for bundled
  hardware profile YAML files
- **CI**: GitHub Actions (`ubuntu-latest`; PCP installed via `apt`)
- **Target platforms**: Linux (primary, CI-validated); macOS (developer primary, Tier 1/2
  only — Tier 3 requires apt PCP)

## Quality Gates

Every pull request and every push MUST pass:

| Gate | Tool | Tier | Condition |
|------|------|------|-----------|
| Linting | ruff | — | Always |
| Type checking | mypy | — | Always |
| Unit tests | pytest | Tier 1 | Always (no PCP needed) |
| Integration tests | pytest | Tier 2 | Always (PCP mocked) |
| E2E tests | pytest | Tier 3 | When `pcp.pmi` importable |
| Profile schema validation | pytest | Tier 1 | All 7 bundled profiles |

A green CI run on every branch is a prerequisite for merging to `main`.

## Governance

- This constitution supersedes all other development practices for pmlogsynth. In case of
  conflict, the constitution prevails.
- **Amendments** require: (1) a written rationale in the PR description, (2) an update to
  this file with version bump and `Last Amended` date, (3) propagation to all dependent
  templates per the Consistency Propagation Checklist in the `/speckit.constitution`
  command.
- **Version semantics**: MAJOR for backward-incompatible principle removals or
  redefinitions; MINOR for new principles or material expansions; PATCH for
  clarifications and wording fixes.
- All implementation plans (plan.md) MUST include a Constitution Check section and MUST
  be evaluated against this document before implementation begins.
- Complexity Tracking tables in plan.md MUST justify any deviation from principles IV
  and V before implementation.
