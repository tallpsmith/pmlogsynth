# OS Metadata Metrics Design

**Issue:** #9 — Include OS metadata metrics in synthetic archives
**Date:** 2026-03-19

## Problem

PCP archives generated by pmlogsynth lack the "log mandatory on once" metadata
metrics that real pmlogger archives contain. Tools like pmview-nextgen that
consume archives cannot determine host OS type, hostname, or hardware topology
from the archive data itself, forcing fallback heuristics.

PCP's default pmlogger configuration (via `pmlogconf`) defines a
`platform/hinv` group (force-include, delta once) that every real archive
contains. pmlogsynth should produce equivalent metadata so synthetic archives
are indistinguishable from real ones at the topology level.

## Scope — Tier 1 Metrics Only

Only the `platform/hinv` group (always-present, force-include):

### String metrics (PM_TYPE_STRING, PM_SEM_DISCRETE)

| Metric | Source |
|--------|--------|
| `kernel.uname.sysname` | `os.sysname` field |
| `kernel.uname.nodename` | `os.nodename` field |
| `kernel.uname.release` | `os.release` field |
| `kernel.uname.version` | `os.version` field |
| `kernel.uname.machine` | `os.machine` field |
| `kernel.uname.distro` | `os.distro` field |

### Integer metrics (PM_TYPE_U32/U64, PM_SEM_DISCRETE)

| Metric | Source |
|--------|--------|
| `hinv.ndisk` | Computed: `len(hardware.disks)` |
| `hinv.physmem` | Computed: `hardware.memory_kb // 1024` (MB) |
| `hinv.pagesize` | `os.pagesize` field (default 4096) |
| `hinv.ninterface` | Computed: `len(hardware.interfaces)` |

`hinv.ncpu` already exists in `CpuMetricModel` and remains there.

## Design Decisions

### OS identity lives in hardware profiles

The `os:` section is added to hardware profile YAML files (bundled and user),
not to the workload profile. A hardware profile describes a complete machine —
its CPUs, disks, NICs, and now its OS identity.

Users override OS fields via the existing `host.overrides:` mechanism:

```yaml
host:
  profile: generic-small
  overrides:
    os:
      nodename: prod-web-03
      distro: "Red Hat Enterprise Linux 9.2"
```

### All os: fields are optional with sensible defaults

| Field | Default |
|-------|---------|
| `sysname` | `"Linux"` |
| `nodename` | value of `meta.hostname` |
| `release` | `"5.15.0-91-generic"` |
| `version` | `"#1 SMP PREEMPT_DYNAMIC"` |
| `machine` | `"x86_64"` |
| `distro` | `"Ubuntu 22.04.3 LTS"` |
| `pagesize` | `4096` |

### Computed hinv.* metrics derive from existing hardware profile data

`hinv.ndisk`, `hinv.physmem`, and `hinv.ninterface` are computed from fields
that already exist in the hardware profile. No new YAML fields needed.

### PM_TYPE_STRING support

`PM_TYPE_STRING` (cpmapi value 6) is added to `pcp_constants.py`. PCP's
`pmiPutValue` already accepts string values — the writer passes `str(value)`
for all types, so no special handling is required.

## Architecture

### New files

- `pmlogsynth/domains/metadata.py` — `MetadataMetricModel` with all metrics
  marked `is_discrete=True`

### Modified files

| File | Change |
|------|--------|
| `pcp_constants.py` | Add `PM_TYPE_STRING` constant |
| `profile.py` | New `OsProfile` dataclass, parse `os:` section, wire into `HardwareProfile` |
| `writer.py` | Add `MetadataMetricModel()` to `self._models` list |
| `cli.py` | Add 10 new metric names to `_ALL_METRIC_NAMES` |
| `pmlogsynth/profiles/*.yml` | Add `os:` section to all 7 bundled profiles |
| `docs/profile-format.md` | Document `os:` section schema |
| `man/pmlogsynth.1` | Update metric count, mention OS metadata |
| `README.md` | Update metric count |

### How it works

The existing discrete metric infrastructure handles everything:

1. `MetadataMetricModel.metric_descriptors()` returns descriptors with
   `is_discrete=True`
2. `MetadataMetricModel.compute()` returns values from `OsProfile` fields and
   computed hardware values
3. Writer's `_write_discrete_sample()` emits them once, one interval before
   the first real sample
4. Per-sample loop skips them via the `discrete_names` set

No changes to the writer's control flow.

## Test Strategy

- **Unit**: `OsProfile` parsing, defaults, override merging
- **Unit**: `MetadataMetricModel` descriptors (count, types, semantics, is_discrete)
- **Unit**: `MetadataMetricModel.compute()` values — string fields, computed integers
- **Integration**: Discrete write includes metadata metrics, string values emitted
- **E2E**: Archive round-trip — `pminfo -f` confirms `kernel.uname.sysname` reads back

## Out of Scope

- Tier 2 metrics (Linux-specific: `kernel.all.boottime`, `network.interface.mtu/speed`)
- Tier 3 metrics (PCP infrastructure: `pmcd.version`, `pmcd.build`)
- Filesystem metrics (`filesys.mountdir`, `filesys.type`, etc.)

These can be added as follow-up issues if needed.
